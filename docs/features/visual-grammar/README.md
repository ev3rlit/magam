# Visual Grammar 고정 가이드 (Magam)

## 1) 문서 목적

본 문서는 "시각 문법 고정"을 Magam에서 실제로 운영 가능한 수준으로 정의한다.

- AI가 생성해도 결과물의 톤과 해석성이 일관되게 유지되도록 한다.
- "예술적 표현"과 "가독성"을 충돌 관계가 아닌 설계 가능한 시스템으로 다룬다.
- 캔버스 오브젝트와 마인드맵 배치 품질을 측정/개선 가능한 규칙으로 표준화한다.

## 2) 시각 문법 고정이란

시각 문법 고정은 다음을 의미한다.

- 같은 의미는 항상 같은 시각 표현을 사용한다.
- 배치 규칙(관계/흐름/강조)이 임의가 아니라 선언된 규칙으로 동작한다.
- 스타일은 "취향"이 아니라 "도메인 의미"에 종속된다.

핵심 공식:

`의미(semantic) -> 시각 어휘(visual lexicon) -> 배치 문장(layout syntax) -> 상태/강조 규칙(state/emphasis)`

## 3) 왜 필요한가

- AI 재현성: 프롬프트가 달라도 결과 품질 편차를 줄인다.
- 팀 협업성: 다른 작성자가 수정해도 스타일 붕괴를 막는다.
- 유지보수성: 기능이 늘어나도 테마/배치 규칙을 확장 가능하게 유지한다.
- 학습 효율: 사용자 시선이 빠르게 구조를 파악한다.

## 4) Visual Lexicon (시각 어휘) 표준

## 4.1 오브젝트 의미 매핑

| 컴포넌트 | 의미(기본 역할) | 기본 시각 규칙 |
|---|---|---|
| `Shape` | 개념/시스템 단위 | 중립 배경 + 명확한 외곽선, 라벨 중심 |
| `Text` | 보조 설명/주석 | 경량 타이포, 배경 없음 |
| `Markdown` | 상세 문서/근거 | 본문 가독성 우선, 폭 제한 |
| `Sticky` / `Sticker` | 주의/결정/핵심 메모 | 고채도 제한 사용(강조 슬롯) |
| `Edge` | 관계(인과/의존/참조) | 방향성 우선, 라벨은 짧게 |
| `Group` | 문맥 경계(모듈/단계) | 약한 컨테이너 배경 + 제목 |
| `MindMap`+`Node` | 계층/주제 확장 | 루트 대비 강화, 깊이별 시각 감쇠 |

## 4.2 색상 의미 슬롯 (예시)

- `neutral`: 기본 노드, 본문, 배경
- `focus`: 현재 작업/선택
- `accent`: 핵심 포인트 (과사용 금지)
- `warning`: 리스크/주의
- `evidence`: 근거/참조

규칙:

- 한 화면에서 `accent`는 10~15% 이내 면적으로 제한
- `warning`은 경고 의미 외 사용 금지
- 동일 의미에 색상 재할당 금지

## 4.3 타이포 계층

- H1(주제), H2(그룹), Body(노드), Caption(엣지/보조)
- 동일 레벨은 폰트 크기/굵기/행간을 고정
- Markdown 블록은 가로폭 상한을 둔다(너비 과확장 방지)

## 5) Layout Syntax (배치 문장) 표준

## 5.1 방향 규칙

- 기본 인과 흐름: 좌 -> 우
- 계층 강조: 상 -> 하 보조
- 역방향 엣지는 예외(피드백 루프 등)로만 허용

## 5.2 그룹/경계 규칙

- 그룹 내부 노드 간격 < 그룹 간 간격
- 그룹 헤더는 좌상단 고정
- 그룹 배경 대비는 노드보다 약하게 유지

## 5.3 밀도 규칙

- 고밀도 영역: 구조 노드 중심
- 저밀도 영역: 설명/주석/시각 휴식 공간
- 캔버스 전체에 "시선 휴식 구간(negative space)"을 의도적으로 배치

## 5.4 마인드맵 규칙

- 루트 노드는 크기/대비 우선
- 깊이가 깊어질수록 시각 강도를 단계적으로 감쇠
- 형제 노드 간 간격은 동일 레벨에서 일관성 유지

## 6) 강조/상태 문법

## 6.1 강조 계층

- 1차 강조: 크기/위치
- 2차 강조: 색상
- 3차 강조: 효과(그림자/스티커)

권장: 한 노드에 2개 이상 강조 수단 동시 사용 금지

## 6.2 상태 표현

- `selected`, `hover`, `editing`, `error`, `resolved` 상태를 전역 공통 스타일로 고정
- 상태 표현은 도메인 스타일보다 우선 적용

## 7) 자동배치 전략 선택 기준

| 목표 | 전략 | 적용 조건 |
|---|---|---|
| 계층 명확성 | `tree` | 부모-자식 중심, 정책/구성도 |
| 균형/아이디어 발산 | `radial` | 브레인스토밍, 주제 확장 |
| 대칭 토론 구조 | `bidirectional` | 장단점 비교, 찬반 구조 |
| 시간 흐름 | `sequence` | 이벤트/프로세스 설명 |
| 수동 의도 반영 | `canvas + anchors` | 저자가 핵심 위치를 지정한 경우 |

## 8) AI 생성 계약 (Prompt Contract)

AI가 따라야 할 최소 제약:

1. 노드 생성 전 의미 타입부터 분류한다.
2. 분류 결과에 따라 컴포넌트를 선택한다.
3. 배치 전략을 명시적으로 선택한다.
4. 강조 슬롯 수를 제한한다.
5. 엣지 라벨은 짧고 동사 중심으로 작성한다.

금지 규칙:

- 의미 없는 랜덤 색상
- 루트/핵심 노드가 주변 노드보다 약하게 보이는 배치
- 그룹 경계 없는 대규모 평면 나열

## 9) 품질 메트릭 (Scorecard)

## 9.1 측정 항목

- `Readability`: 탐색 난이도, 텍스트 대비, 라벨 길이
- `Structural Fidelity`: 의미 구조와 배치 일치도
- `Aesthetic Balance`: 여백/정렬/리듬 균형
- `Editability`: 수정 시 영향 범위 최소화(그룹/앵커 안정성)

## 9.2 예시 지표

- Edge crossing 수
- Node overlap 수
- 역방향 엣지 비율
- 루트에서 평균 hop depth
- 강조 토큰 사용 비율

## 10) 운영 가드레일

- 디자인 토큰은 semantic 우선 사용
- 신규 컴포넌트는 시각 역할을 먼저 등록한 후 개발
- PR 체크리스트에 다음 항목 추가:
  - 의미-표현 매핑 위반 없음
  - 배치 전략 선택 근거 있음
  - 상태 스타일 표준 준수

## 11) 구현 권장 구조 (코드 관점)

- `@magam/shared`: 시각 문법 스키마(타입 + 검증)
- `app/store`: 현재 문서에 적용된 grammar profile 상태
- `app/hooks`: 레이아웃 전략 선택기 + 품질 점수 계산기
- `app/components`: profile 기반 렌더 스타일 적용

권장 패턴:

- FP: 점수 계산, 배치 변환, 의미 분류
- OOP: 전략 객체(`LayoutStrategy`), 프로필 레지스트리
- Design Pattern: Strategy, Factory, Policy Object

## 12) 단계별 도입 계획

### Phase 1: 문법 정의

- 오브젝트 의미 매핑 고정
- 색/타입/상태 슬롯 정의

### Phase 2: 자동 검증

- 점수 계산기 도입
- 최소 품질 임계치 도입

### Phase 3: AI 루프

- 생성 -> 평가 -> 재배치 자동 루프
- 프리셋 프로필별 결과 비교

## 13) 오픈 질문

- export 산출물에서 현재 테마를 고정할지 선택 가능으로 둘지
- 마인드맵 깊이별 감쇠 규칙을 폰트/색/패딩 중 어디에 우선 적용할지
- 예술성 확장을 위해 허용할 랜덤성의 범위를 어떻게 제한할지
